# 🚨 阶段四详细设计：边界安全、免责防御与上线方案

## 1. 核心技术安全策略：如何做到“绝对合规”？

国内对内网穿透最敏感的监管点在于：**防止用户绕过国家防火墙（监管）私自搭建未备案的 Web 网站（HTTP/HTTPS 流量）**。
既然你的产品定位是“只做极简终端敲字”，我们就必须在底层把一切非 SSH 流量彻底熔断。

### A. 控制面策略：Headscale 端口ACL熔断（云端降维打击）

Tailscale/Headscale 拥有非常强大的 **ACL（访问控制列表）** 策略引擎。你需要让 Claude Code 在云端控制中心（`cloud-server/`）的 Headscale 配置文件中，**硬编码强行注入**以下全局 ACL 规则：

```json
// ACL 规则伪代码：只放行 22 端口，其余全量丢弃
{
  "acls": [
    {
      "action": "accept",
      "src": ["*"],
      "dst": ["*:22"] // 核心防御：只允许访问任意内网机器的 22 端口（SSH）
    }
  ],
  "ssh": [
    {
      "action": "accept",
      "src": ["*"],
      "dst": ["*"],
      "users": ["*"]
    }
  ]
}

```

* **效果**：有了这几行硬编码，即使硬核用户试图修改客户端配置，把本地的 80 端口（网站）或 3389 端口（Windows远程桌面）映射进来，当流量到达虚拟网卡时，也会被 WireGuard 底层**直接无情丢弃**。这就从技术上向网安证明了：“我这个工具纯粹是生产力终端，技术上根本不具备搭网站的能力。”

### B. 数据面策略：国内网安合规与实名追溯（Audit Log）

既然你要给少数会员提供高质量的连接服务，根据国内《网络安全法》，你需要记录必要的连接日志，用于在发生极端事件时进行免责举证。

* **你的策略**：在 `cloud-server/` 中，当用户成功建立 P2P 握手或发起轮询时，后端必须在 SQLite 中记录审计日志（不记录用户敲字的内容，只记录行为）：
```text
[TIMESTAMP] 用户ID: 10023 (手机号: 138xxxx8888) -> 激活设备: Node_WSL2 (内网IP: 100.64.0.3)

```


* **效果**：一旦某个会员在公司内网搞破坏被大厂网安追踪，你作为平台方可以秒级交出日志证明“管道是合规的，违法的是具体的实名自然人”，实现完美的法律免责。

## 2. 商业化上线准备：极客专属的“自备中转（DERP）”机制

我们在总纲里提到，为了不承担高昂的公网服务器带宽成本，我们要把“中转”的权利和成本交回给用户。
你需要在 `linux-client/` 目录下，让 Claude Code 编写一个**私有化 DERP 中转站一键部署脚本**。

* **原理**：Tailscale 生态中的中转站叫做 **DERP 服务器**。如果用户在公司打洞失败，流量就需要通过一个 DERP 节点中转。
* **你的做法**：在 App 的高级设置里，提供一个按钮叫“一键生成我的私有中转脚本”。
用户点击后，App 生成一段如下命令，用户复制到他们**自己买的**任意国内轻量云服务器上运行：
```bash
# 自动化下载并运行个性化 DERP 节点的脚本
curl -fsSL https://api.nextconnect.com/scripts/install-derp.sh | sh -s -- --secret=用户专属密钥 --port=443

```


* **绝妙之处**：用户的流量完全在他们自己的公司电脑和他们自己的云服务器之间流转。**你的主服务器只负责给他们牵线搭桥（0带宽消耗），用户自己承担了所有的带宽和合规成本，你躺着净赚会员费。**
